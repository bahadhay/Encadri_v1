⏺ I found the root cause! The issue is that audioOptions: { muted: false } doesn't actually start capturing audio - it just
   sets a flag. Azure Communication Services requires an ACTIVE media stream (actually capturing video or audio) to
  establish the peer connection.

  When you join with:
  - Camera OFF + Mic "unmuted": No active stream → Connection doesn't establish (call state stays "None")
  - Camera ON + Mic muted: Active video stream → Connection establishes ✅

  The problem is that setting muted: false means "don't mute IF audio starts", but it doesn't actually start capturing
  audio. Azure SDK doesn't have a LocalAudioStream like it has LocalVideoStream - audio is automatically managed once the
  call connects, but you need video or an already-connected call first.

     ↓
  Frontend calls API: POST /videocall/notify-meeting-started/{meetingId}

     ↓

  Student waits in waiting room (sees countdown + spinner)
     ↓
  SignalR connection receives "MeetingStarted" event
     ↓
  Frontend checks: Is this my meeting?
     ↓
